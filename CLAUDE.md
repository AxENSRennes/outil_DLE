## Security & Defensive Coding Rules

### Fail-closed by default
- Any security-critical operation (logout, session clear, token revocation) MUST be in a `finally` block so it executes even if surrounding code (e.g. audit writes) raises.
- When implementing a fail-closed pattern on one function, audit all sibling functions that perform similar operations and apply it consistently.

### Pattern consistency
- When a security or architectural pattern is applied to one function, systematically check and apply it to all analogous functions in the same module.
- Before finishing a feature, review all new functions as a group — not just individually.

### Audit trail immutability
- Django admin classes for audit/log models MUST override `has_add_permission`, `has_change_permission`, and `has_delete_permission` to return `False`. Audit records are append-only.

### Credentials and secrets in admin
- Any model field that stores hashed or sensitive values (PINs, passwords, tokens) MUST be excluded from admin forms or marked as `readonly_fields`. Never allow plain-text editing of hashed fields.

### Trust boundaries on request data
- Never blindly trust client-supplied headers (e.g. `X-Forwarded-For`) for security decisions. Document trust assumptions when using such headers.
- If a value derived from an untrusted source is stored (e.g. in audit metadata), document that it is advisory/best-effort.

### General
- Run `make check` before considering any feature complete. All linters, type checks, tests, and security scans must pass.

## API Contract Rules

### No generic DictField/ListField for structured data
- In DRF serializers, NEVER use `serializers.DictField()` or `serializers.ListField()` for fields with a known structure. Always create a typed nested serializer instead.
- `DictField` / `ListField` are only acceptable for truly dynamic data (e.g. freeform `data_json`, `meta_json`).
- This ensures the OpenAPI schema generated by drf-spectacular is precise and that DRF validates outgoing payloads.

### Selectors must normalize JSON blob outputs
- When a selector reads from an untyped JSON blob (e.g. `snapshot_json`), it MUST explicitly extract and default every field that the corresponding serializer expects.
- Do NOT pass raw `dict.get("key", {})` to a nested serializer — build the complete dict with all required keys and safe defaults.
- This prevents incomplete data from reaching the serializer and the frontend.

## Frontend Component Rules

### Use shadcn/ui — do not reinvent primitives
- This project uses **shadcn/ui** (configured in `frontend/components.json`, components install to `@/shared/ui`).
- Before writing any UI component from scratch, check if a shadcn/ui component exists for that purpose (Badge, Button, ScrollArea, Sidebar, Card, Tooltip, Sheet, etc.).
- Install missing shadcn/ui components with `npx shadcn@latest add <component>` from the `frontend/` directory rather than hand-coding equivalents.
- Domain-specific components (e.g. StepStatusBadge, BatchHeader) should **compose** shadcn/ui primitives, not duplicate their functionality.
- Only write raw Tailwind markup for truly novel layouts that have no shadcn/ui or Radix primitive equivalent.

## Naming Conventions

### Python
- Modules, functions, variables: `snake_case` — classes: `PascalCase`
- Timestamp fields always end in `_at` (`created_at`, `updated_at`, `signed_at`)
- Boolean fields always start with `is_` (`is_active`, `is_required`, `is_deleted`)
- Foreign keys use `<entity>_id` (`batch_id`, `mmr_version_id`, `site_id`)
- Table names: `snake_case` plural (`batches`, `batch_steps`, `review_events`)

### TypeScript
- Variables/functions: `camelCase` — types/interfaces/schemas/components: `PascalCase`
- File naming: `kebab-case` for non-component modules, `PascalCase` for React component files
- Path alias `@/*` resolves to `./src/*` — always use it instead of relative paths

### API boundary
- API JSON payloads use `snake_case` (aligned with Django/DRF) — no camelCase translation at the boundary
- REST resources use plural nouns: `/api/v1/batches`, `/api/v1/mmr-versions`
- URL path segments use `kebab-case`, query parameters use `snake_case`
- All endpoints under `/api/v1` with explicit versioning
- Dates/datetimes use ISO 8601 strings with timezone
- Use explicit `null` for absent data — empty string is never a substitute for missing regulated data

### Domain language
- Always use canonical domain terms — never synonyms or generic names (`item`, `record`, `data`) when the domain object has a specific name (e.g. `batch`, `review_event`)
- Internal domain events use past-tense business naming (e.g. `batch_started`, `step_signed`)
- Frontend state names must match backend exactly — no renaming or mapping

## Architecture Boundaries

### Backend — modular monolith
- Each domain in `backend/apps/<domain>/` with sub-packages: `api/`, `domain/`, `selectors/`, `tests/`
- Regulated business rules live in `domain/` — never in DRF serializers or API views
- `domain/` must not depend on `api/`; `api/` may call `domain/` and `selectors/`
- `backend/shared/` provides generic technical primitives — must not encode feature business rules

### Frontend — three layers
- `app/` (composition, routing, providers, layouts), `shared/` (primitives, API client, types, utils), `features/` (workflow-specific UI)
- Feature-domain organization: `features/execution/`, `features/pre-qa-review/`, `features/quality-review/`, `features/auth/`, etc.
- Features must not deep-import each other — cross-feature coordination through `app/` wiring or shared contracts
- `frontend/src/shared/` must remain dependency-light
- Route-level code splitting; no SSR (authenticated SPA)
- TanStack Query for all server state — no Redux, Zustand, or global state store
- Local UI state stays in React primitives (`useState`, `useReducer`)
- React Hook Form + Zod for complex regulated forms

## Workflow & Domain Integrity

- Workflow transitions are explicit backend action endpoints — no arbitrary PATCH on status fields
- Backend is the source of truth for all regulated workflow state — frontend never reconstructs state from raw audit history
- Signatures are business events, not UI widgets — each signature stores: signer identity, server timestamp, signature meaning, related batch step, signed-state reference
- Step-up re-authentication is mandatory before any signature action, even when session is active
- Review flags (`changed_since_review`, `changed_since_signature`, `review_required`) are stored/derived in backend — never inferred only from audit trail in frontend
- Batch instances freeze a template snapshot at creation — later template changes must never mutate in-progress records
- Workflow state must be exposed explicitly in API responses — never force clients to infer state from raw audit history

## Shared Workstation Model

- PIN-based workstation identification — Django session cookies are the only browser auth authority (no JWT/token auth)
- Explicit lock clears authenticated authority entirely
- `switch_user` replaces the prior authenticated session without losing batch context
- Identify and signature re-auth endpoints have dedicated throttle scopes
- Failed PIN attempts and identity actions must emit audit events
- Persist draft field values on blur so user switches lose at most one field interaction

## API Response Conventions

- Collections return direct resource lists or paginated DRF structures — no custom wrappers
- Single resources return the representation directly
- Errors use problem-details JSON format with stable machine-readable codes and user-safe messages
- Validation errors must be structurally predictable and field-addressable
- OpenAPI generated via drf-spectacular as the canonical API contract

## Testing Conventions

- Backend tests co-located in each app: `backend/apps/<domain>/tests/` — separate modules for API, domain, and integration coverage where needed
- Cross-app and contract tests in `backend/tests/` (integration/, contract/)
- factory-boy for readable fixtures — no JSON fixtures or hard-coded data in tests
- Frontend tests co-located: `src/**/*.test.ts`, `src/**/*.test.tsx`
- Testing Library for component tests — user-event for interaction simulation
- Every behavior change must include or update tests in the same domain area
- Workflow transitions must be tested through backend action endpoints — never via direct model mutation
- Signature, review, and release paths require thorough failure-case coverage

## Frontend UX Invariants

- Validation UX: "reward early, punish late" — validate on blur, not on keystroke; defer empty-required errors until completion attempt
- Progressive auto-save with mandatory visible feedback (saving/saved/failed)
- Keyboard-first: all critical operator and reviewer actions must be keyboard-accessible
- Explicit loading/error states for every mutation — never auto-retry sensitive writes without user intent

## Migration Policy

- Migrations must be additive expand/contract only — never destructive schema changes in the same deployment window (3x8 plant constraint)

## Anti-Patterns to Avoid

- Mixing `camelCase` and `snake_case` in API payloads
- Putting signature business rules inside React components
- Inconsistent response shapes (`{ success: true, data: ... }` on one endpoint and raw JSON on another)
- Using generic names (`item`, `record`, `data`) when the domain object has a specific name
- Inventing parallel abstractions for workflow state, signatures, or review semantics
- Arbitrary PATCH on status fields — always use dedicated action endpoints
- Hiding review-relevant state changes in audit logs instead of exposing them explicitly in API responses
